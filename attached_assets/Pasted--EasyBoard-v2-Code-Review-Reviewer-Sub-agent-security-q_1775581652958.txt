### 🔴 BUG 1: Missing `status` column on `meetings` table

**File:** `lib/db/src/schema/meetings.ts`

The schema defines `meetingsTable` with no `status` column. However, the meetings route has:

```typescript
// artifacts/api-server/src/routes/meetings.ts — PATCH handler
if (status != null) updates.status = status;  // ← will cause DB error
```

And the meetings listing assumes a `status` field for scheduling logic. This will crash at runtime when an admin tries to update meeting status.

**Fix:** Add `status: text("status", { enum: ["scheduled", "in_progress", "adjourned", "cancelled"] }).default("scheduled")` to `meetingsTable`.

### 🔴 BUG 2: Missing `UNIQUE(entity_type, entity_id, person_id)` on `access_control` table

**File:** `lib/db/src/schema/accessControl.ts`

```typescript
export const accessControlTable = pgTable("access_control", {
  // ...
}, (t) => ({
  uniq: unique().on(t.entityType, t.entityId, t.personId),  // ← defined in Drizzle...
}));
```

The constraint IS defined in the schema, BUT Drizzle ORM's `text({ enum: [...] })` does NOT create a PostgreSQL ENUM type — it just stores as `text` with TypeScript-only type narrowing. This means duplicate access grants could be inserted at the DB level if someone bypasses the application. **Low risk** since all writes go through the app layer, but not enforced at DB level.

## 5. Security Issues

### 🔴 CRITICAL: Hardcoded JWT Secret Fallback

**File:** `artifacts/api-server/src/lib/auth.ts`, line ~11

```typescript
const JWT_SECRET = process.env.SESSION_SECRET || "easyboard-secret-key-change-in-prod";
```

If `SESSION_SECRET` is not set, the app runs with a predictable secret. An attacker who obtains the source code could forge JWTs for any user, including admin.

**Fix:** Fail fast if the secret is missing:
```typescript
const JWT_SECRET = process.env.SESSION_SECRET;
if (!JWT_SECRET) throw new Error("SESSION_SECRET environment variable is required");
```

### 🟡 MEDIUM: Wildcard CORS

**File:** `artifacts/api-server/src/app.ts`, line ~19

```typescript
cors: { origin: "*", methods: ["GET", "POST"] }
```

`origin: "*"` allows any website to make API requests on behalf of users. For a board portal handling confidential governance data, this is a significant risk.

**Fix:** Restrict to known frontend origin(s) via `process.env.ALLOWED_ORIGIN`.

### 🟡 MEDIUM: No Rate Limiting

No rate limiting anywhere. The AI `callAI` endpoint is unprotected against abuse. A malicious actor could exhaust the Anthropic API quota.

### 🟡 MEDIUM: Default Seed Password in Source

**File:** `artifacts/api-server/src/seed.ts`, line ~8

```typescript
const PASSWORD = "Meridian2024!";
```

All 20 seeded users get this same password. If source code is leaked, all seeded accounts are compromised. The password should come from `process.env.SEED_PASSWORD` with a strong random fallback.

### 🟢 LOW: Silent AI Errors in Fire-and-Forget Paths

**File:** `artifacts/api-server/src/routes/documents.ts`, line ~90:
```typescript
} catch {
  // AI failed silently — document already stored
}
```

**File:** `artifacts/api-server/src/routes/tasks.ts`, line ~117:
```typescript
} catch {
  // Evidence stored, AI review skipped
}
```

No logging of AI failures. Difficult to debug production issues. Should at minimum log at `warn` level.

### 🟢 LOW: XSS Risk in AI Search Links

**File:** `artifacts/api-server/src/routes/ai.ts`

The SEARCH mode extracts link placeholders `[entityType:entityId:title]` from AI text and renders them as links. The `title` extracted from AI output is rendered directly in HTML without sanitization:

```typescript
const cleanAnswer = answer.replace(linkRegex, (_, type, id, title) => {
  sources.push({ entityType: type, entityId: id, title });
  return title;  // ← title rendered as HTML text content, not innerHTML
});
```

The `title` goes into the DOM as text content (not HTML), so XSS is mitigated. However, if this data is used elsewhere in the frontend in an `innerHTML` context, it could be a problem.

## 11. Additional Findings

### 🟡 `close_task` Bug in Pending Actions Handler

**File:** `artifacts/api-server/src/routes/pendingActions.ts`, `executeAction("close_task", ...)`:

```typescript
case "close_task": {
  const { task_number, taskId, reasoning } = actionData as any;
  let resolvedTaskId = taskId;
  if (!resolvedTaskId && task_number) {
    const [t] = await db.select().from(tasksTable);  // ← ignores task_number!
    resolvedTaskId = t?.id;
  }
```

The `task_number` parameter is read but never used in the query. The query selects ALL tasks and takes the first one. If `taskId` is not provided, it will close the wrong task.

**Fix:** Filter by `taskNumber`:
```typescript
const [t] = await db.select().from(tasksTable)
  .where(eq(tasksTable.taskNumber, task_number))
  .limit(1);
```

### 🟡 `migratePeopleTitles` Is a Side-Effect in `seed()`

**File:** `artifacts/api-server/src/seed.ts`

`migratePeopleTitles()` is called at the end of both the seeding path AND the "already seeded" skip path. This is fine but confusing. The function name implies migration, not seed maintenance.

### 🟡 `proxyHolderId` in Attendance Table

The `attendance` table has a `proxyHolderId` column, but no route implements proxy voting. The column is orphaned.

### 🟡 Board Seed Membership Discrepancy

In the seed, the Finance & Audit Committee is listed with 6 members including Laura Martinez (observer), but the prompt specified "Finance & Audit Committee (FAC) — 4 members + 2 observers." The seed has 4 members + 2 observers = correct count, but the BoD has 7 voting members + Secretary + 2 observers = 10 total. The prompt specified "Board of Directors (BoD) — 7 voting members + Secretary + 2 observers" = 10. Correct. ✅

### 🟢 Approval Rule Type: `weighted` Not Enforced

`approvalRuleWeightsTable` exists and stores weights, but `evaluateByRule()` in `votes.ts` does not apply weights to the vote count. Weighted voting is stored but not calculated. Low priority — the infrastructure exists and can be activated later.
