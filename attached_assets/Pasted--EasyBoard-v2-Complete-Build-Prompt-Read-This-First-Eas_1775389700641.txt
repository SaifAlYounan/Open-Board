# EasyBoard v2 — Complete Build Prompt

## Read This First

EasyBoard v2 is an **AI-native board management portal**. The AI IS the product. Every document uploaded is parsed, classified, and cross-referenced by Claude. The AI proposes the next action. The Secretary approves or overrides. Everything else follows.

**The demo IS the product.** Upload a document. Watch the AI process it. Approve the actions. See the results appear across all interfaces. The system starts empty — no pre-seeded meetings, votes, or documents.

This is NOT a rebuild of v1. It is a reimagining of the core interaction model.

**CRITICAL BUILD RULES:**
1. Build in 4 phases. Complete each phase fully and test before starting the next.
2. Do NOT skip steps or reorder phases.
3. When a test checkpoint says "TEST:", stop and verify before proceeding.
4. If something breaks, fix it before moving on — do NOT accumulate tech debt.

---

## The Core Concept: 5 Layers

```
Layer 0 — Document Brain     (invisible infrastructure)
Layer 1 — Secretary          (AI copilot, approval queue)
Layer 2 — Board Member       (zero friction, AI search)
Layer 3 — Management         (tasks + evidence)
Layer 4 — Observer           (read-only + comments)
```

**Layer 0 — Document Brain** is the invisible foundation. Every uploaded file goes through Claude classification. The AI returns structured JSON: `document_type`, `extracted_data`, `proposed_actions`. This is a trained classification system with clear rules for each document type.

**Layer 1 — Secretary Interface** is an AI copilot. Upload a document → AI tells you what it will create and why → you approve, edit, or cancel. Natural language commands: "Schedule a BoD meeting for April 15 with GreenTech on the agenda" → AI creates it. Full CRUD override always available.

**Layer 2 — Board Member Interface** is zero friction. One dashboard: pending votes, minutes to sign, next meeting. AI search bar: "When did we discuss Kazakhstan?" → AI finds the answer with source links. Vote in 2 clicks. Sign in 1 click. No admin controls anywhere.

**Layer 3 — Management Interface** is task-centric. My Tasks front and center with status, deadlines, and source links. Evidence upload per task with AI review. Read-only view of board decisions.

**Layer 4 — Observer Interface** is management minus task upload. Read-only access. Can comment on minutes in review. AI search.

- Tables are ordered to satisfy all foreign key dependencies — execute in order.
- `agenda_items.vote_id` references `votes(id)` — votes table MUST be created before agenda_items.
- `ai_classification` on `documents` stores the full Claude response as JSONB.

- All `approval_rule_*` junction tables use `ON DELETE CASCADE` from the rule.

## Access Control — Default Grant Logic

**CRITICAL:** When any entity is created (via AI approval or manual CRUD), the backend MUST automatically create `access_control` rows using these rules:

| Entity Created | Who Gets Access |
|---|---|
| Meeting | All members of the meeting's board (via `board_memberships`) + all observers of that board |
| Vote | All voting members of the vote's board + all observers of that board |
| Minutes | All members + observers of the minutes' meeting's board |
| Task | The assignee + all admin users + all observers of the source board |
| Document | All members + observers of the document's board. If no board assigned, admin only until Secretary assigns. |

**Implementation:** Create a helper function `grantDefaultAccess(entityType, entityId, boardId)` that queries `board_memberships` for the given board and inserts `access_control` rows. Call this function in every entity creation path (both AI-approved and manual CRUD).

**Secretary (admin role)** always has implicit access to everything — queries should include `WHERE person.role = 'admin' OR EXISTS (SELECT 1 FROM access_control ...)`.

## AI Configuration

### Model Selection

Use `claude-opus-4-20250514` for ALL AI calls. Classification accuracy and nuanced reasoning (confidentiality detection, evidence matching, vague deadline computation) are critical for a board management tool — this is a governance product where a misclassification has real consequences. Opus is slower and more expensive than Sonnet, but the accuracy tradeoff is worth it for a demo and for production at low-to-moderate volume.

**For high-volume production deployments:** If API costs become a concern at scale, switch SEARCH, SUGGEST, and COMMAND modes to `claude-sonnet-4-20250514` (these are simpler tasks where Sonnet performs well). Keep CLASSIFY and REVIEW on Opus.

### Token Limits

| AI Mode | Max Input (document text) | Max Output Tokens |
|---|---|---|
| CLASSIFY | Truncate document text to 12,000 tokens (~48,000 chars). If longer, extract first 6,000 + last 6,000 tokens. | 4,096 |
| COMMAND | N/A (user input is short) | 2,048 |
| SEARCH | Search results capped at 8,000 tokens | 2,048 |
| REVIEW | Evidence text truncated to 8,000 tokens | 2,048 |
| SUGGEST | Pending items capped at 4,000 tokens | 1,024 |

### Error Handling

**CRITICAL:** Every AI call MUST be wrapped in try/catch with these fallbacks:

```javascript
async function callAI(mode, systemPrompt, userContent) {
  // 1. Check if API key exists
  if (!process.env.ANTHROPIC_API_KEY) {
    return { error: 'no_api_key', message: 'AI features require configuration.' };
  }

  try {
    const response = await anthropic.messages.create({
      model: 'claude-opus-4-20250514',
      max_tokens: TOKEN_LIMITS[mode],
      system: systemPrompt,
      messages: [{ role: 'user', content: userContent }]
    });

    const text = response.content[0]?.text;
    if (!text) {
      return { error: 'empty_response', message: 'AI returned no content.' };
    }

    // 2. Parse JSON — strip markdown fences if present
    const clean = text.replace(/```json\n?|```\n?/g, '').trim();
    const parsed = JSON.parse(clean);
    return { success: true, data: parsed };

  } catch (err) {
    // 3. Handle specific error types
    if (err.status === 429) {
      return { error: 'rate_limited', message: 'AI is busy. Try again in a moment.' };
    }
    if (err.status === 401) {
      return { error: 'invalid_key', message: 'API key is invalid. Check Settings.' };
    }
    if (err instanceof SyntaxError) {
      // AI returned non-JSON — store raw response, let Secretary handle manually
      return { error: 'parse_error', message: 'AI response could not be parsed.', raw: text };
    }
    return { error: 'unknown', message: 'AI service unavailable. Create entities manually.' };
  }
}
```

**In the UI:** When AI fails, show a toast with the error message and a "Create Manually" button that opens the standard CRUD form. The system must NEVER block the Secretary from working because the AI is down.

**For CLASSIFY errors:** Store the document in the `documents` table with `ai_classification = NULL`. Show the document in the Secretary's document library with a badge: "⚠️ AI classification failed — classify manually." Provide a "Retry Classification" button.

## AI Mode: CLASSIFY (Document Upload)

Call Claude with: `system: AI_BRAIN_PROMPT + CLASSIFY_PROMPT`, `user: DATABASE_CONTEXT + DOCUMENT_TEXT`

**CLASSIFY_PROMPT (append to system prompt):**

```
MODE: CLASSIFY

A document has been uploaded. Analyze it and return ONLY valid JSON — no preamble, no markdown fences, no explanation outside the JSON.

Return this exact structure:
{
  "document_type": "draft_minutes" | "resolution" | "financial_report" | "legal_opinion" | "evidence" | "meeting_agenda" | "general",
  "confidence": 0.0-1.0,
  "extracted_data": { /* varies by type — see rules below */ },
  "proposed_actions": [
    {
      "action_type": "create_minutes" | "create_vote" | "create_meeting" | "create_task" | "close_task" | "attach_to_meeting" | "flag_confidential",
      "description": "human-readable description of what this action will do",
      "details": { /* action-specific data needed to create the entity */ }
    }
  ]
}

CLASSIFICATION RULES:

## 1. DRAFT MINUTES
How to recognize: Contains "MINUTES OF THE", "Present:", "Apologies:", numbered agenda items, "RESOLVED THAT", "ACTION:", "The meeting was adjourned at".
Extract: meeting_date, board_name, attendees[], agenda_items[], action_items[{assignee, task, deadline}], resolutions[], confidential_passages[]
Action item triggers — look for:
  - "ACTION: [Person] to [task] by [date]"
  - "[Person] was tasked with..."
  - "The Board directed [Person] to..."
  - "[Person] to submit/prepare/complete [deliverable] by [deadline]"
  Each must have: a named person, a clear deliverable, and a deadline. If the deadline is vague ("by next quarter"), calculate the actual date.
Confidentiality triggers — flag any paragraph containing:
  - Pricing, discount percentages, negotiation positions
  - Individual compensation or performance data
  - Legal privilege ("legal advice", "attorney-client", "without prejudice")
  - Acquisition valuations before public announcement
Propose: create_minutes + create_task (one per action item) + flag_confidential (one per flagged passage)

## 2. RESOLUTION / REQUEST FOR APPROVAL
How to recognize: Contains "RESOLVED THAT", "IT IS HEREBY RESOLVED", "REQUEST FOR BOARD APPROVAL", formal resolution language.
Extract: resolution_text (the full "RESOLVED THAT..." paragraph), board_name, subject, suggested_deadline
Propose: create_vote — in your description, tell the Secretary: "This can be put to the Board as a meeting vote (if a meeting is scheduled) or a circulation vote (asynchronous with deadline). Which do you prefer?"
If the document mentions a specific meeting: link the vote to that meeting's agenda.

## 3. FINANCIAL REPORT
How to recognize: Contains "Financial Statements", "Revenue", "EBITDA", "Balance Sheet", "Cash Flow", period references (Q1 2025, FY 2024).
Extract: period, key_figures[{label, value}], summary
Propose: attach_to_meeting — check DATABASE_CONTEXT for upcoming meetings of the relevant board. If a meeting exists, propose attaching. If no meeting exists, just store it with no proposed actions.

## 4. EVIDENCE / DELIVERABLE
How to recognize: A document that appears to be a deliverable or proof of completion — a report, assessment, permit, certificate, compliance document.
How to match: Compare the document's content against the OPEN TASKS in DATABASE_CONTEXT. Look for keyword overlap between document summary and task title/description.
Propose: close_task — include reasoning: "This document appears to satisfy [task_number] because [reasons]."
If no task matches: classify as "general" instead. Do not force a match.

## 5. MEETING AGENDA
How to recognize: Contains "AGENDA", numbered items with types like "For Information", "For Discussion", "For Decision/Approval", a meeting date and time, a board/committee name.
Extract: meeting_date, board_name, location, agenda_items[{title, type, description}]
Propose: create_meeting with the full agenda.

## 6. LEGAL OPINION
How to recognize: From a law firm, contains legal analysis, references to laws/regulations, "In our opinion", case citations.
Extract: summary, key_conclusions[], regulatory_deadlines[]
Propose: attach_to_meeting if it relates to an upcoming agenda item. Always propose flag_confidential.

## 7. GENERAL
Catch-all for anything that doesn't fit the above. Extract: summary, key_topics[], entities_mentioned[]. Propose no actions beyond storing.
```

Answer concisely. Include entity references as links: [Meeting: BoD Q2 2025], [Vote: RES-BOD-2024-002], [Document: Kazakhstan Investment Memo].

If no search results match the question, say: "I couldn't find anything matching your question in the documents you have access to."
```

## Database Context Injection

**CRITICAL:** Before every CLASSIFY and COMMAND call, the backend queries the database and injects current state into the USER message (not the system prompt):

```
CURRENT DATABASE STATE:

Boards: [list all boards with abbreviations and member counts]

Open tasks (for evidence matching):
[...all tasks with status != 'done', showing: task_number, title, assignee name, due_date]

Upcoming meetings (for document attachment):
[...all meetings with status = 'scheduled', showing: title, date, board abbreviation, agenda item count]

Open votes:
[...all votes with status = 'open', showing: resolution_number, title, board abbreviation, deadline, votes cast / eligible]

People:
[...all people with name, role, title — for assignee matching]

This context is queried fresh on each call. Without it, the AI cannot match evidence to tasks, route documents to boards, or link entities. Build a helper function `getDatabaseContext()` that runs these queries and formats the output.

### Phase 2 — Document Brain (the core AI pipeline)

9. File upload endpoint: `POST /api/documents/upload` using multer. Accept PDF, DOCX, TXT only. Max 10MB. Store files in `/uploads/` directory.

13. Secretary approval flow:

    - **Edit:** Opens a modal with the proposed data pre-filled. Secretary edits. On save, updates `action_data` and then executes as if approved. Sets status = 'modified'.
    - **Reject:** Sets `pending_actions.status = 'rejected'`. No entity created.
14. Natural language command bar (Secretary only): Text input at top of dashboard. `POST /api/ai/command` → returns proposed actions → adds to pending_actions queue → same approval flow.

## Approval Rules — Per Vote Configuration

When creating a vote (via AI or manually), the Secretary configures an approval rule:

**Presets (one click):**
- Unanimous — all eligible voters must approve
- Simple Majority — more than 50%
- Two-Thirds — ≥ 66.7%
- Three-Quarters — ≥ 75%

**Custom builder (expandable panel):**
- Minimum approvals: number input (e.g., "at least 4 of 7")
- Required voters: multi-select dropdown (people who MUST approve)
- Weighted votes: toggle, then assign weights per person
- Recused members: multi-select (excluded from count, cannot vote)
- Quorum: number input (minimum participants)
- Deadline behavior: dropdown (Lapse / Auto-extend 7 days / Notify Secretary)

After configuring, show a plain-English summary:
> "This resolution passes when at least 4 of 6 eligible voters approve, including Nadia Petrov. James O'Brien is recused. Quorum: 5. Lapses if unresolved by deadline."

## Critical Rules (non-negotiable)

1. **Vote ONCE only.** `UNIQUE(vote_id, person_id)` constraint. Backend returns `409 Conflict` on duplicate vote. Frontend removes vote buttons from DOM after voting.
2. **No Abstain.** Exactly 4 vote options: Approved, Approved with Comments, Not Approved, Not Approved with Comments.
3. **Admin buttons absent from DOM for non-admins.** Not greyed out, not hidden with CSS — not rendered at all. Use conditional rendering in React: `{user.role === 'admin' && <AdminButton />}`.
4. **Permissions are live.** Every query joins against `access_control` (or checks `user.role === 'admin'`). If access is revoked, the entity disappears from all views including AI search results.
5. **SHA-256 on signatures.** `hash = SHA-256(minutes_content + signer_name + ISO_timestamp)`. Display the hash in the signature block.
6. **App works without ANTHROPIC_API_KEY.** If the key is absent, all AI features show a blue info banner — no crashes, no broken pages, no console errors.
7. **No pre-seeded data beyond org/boards/people/memberships.** The demo IS uploading a document and watching the AI work.
8. **Comments only for minutes.** No inline edits, no deletions, no colored highlights, no `String.replace()` on Tiptap content. Comments are anchored to `original_text` and rendered in a sidebar.
9. **Secretary approval is mandatory.** The AI proposes. The Secretary disposes. No action executes without approval. Every AI-proposed action goes through `pending_actions`.
10. **Signature animation is CSS only.** `opacity: 0 → 1` + `transform: scale(0.8) → scale(1)` over 0.5s. No letter-by-letter JavaScript.
11. **PDF output via `window.print()`.** HTML page with `@media print` styles. No puppeteer, no pdf-lib.
12. **Font bundling.** Inter and Dancing Script served from `/public/fonts/`. Zero external font requests.

## Production Deployment Notes

When moving from Replit to production:

1. **Database:** Replace Replit's PostgreSQL with a managed instance (e.g., Neon, Supabase, RDS). The schema is standard PostgreSQL — no Replit-specific features used.
2. **File storage:** Replace `/uploads/` local directory with S3-compatible object storage. The `file_path` column in `documents` and `task_evidence` stores the path — update the upload handler to write to S3 and store the S3 key.
3. **Auth:** JWT is production-ready. Add token refresh and short expiry (15 min access token + 7 day refresh token) when deploying.
4. **Environment:** All config is via environment variables — no hardcoded values.
5. **CORS:** Configure for your domain. Replit's default CORS is permissive.
6. **Rate limiting:** Add express-rate-limit on auth endpoints and AI endpoints in production.
