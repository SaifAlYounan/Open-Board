# EasyBoard v2 — Definitive Bug Report & Fix Prompt

## READ THIS FIRST

This is based on exhaustive end-to-end browser testing across all 4 roles (Secretary, Board Member, Management, Observer). The AI pipeline is excellent — do NOT touch the AI prompts, brain system prompt, or classification logic. The bugs are in the data layer, approval execution, and missing UI features.

**CRITICAL ARCHITECTURE NOTE:** This must be ONE app with ONE database. All 4 roles (/secretary, /board, /management, /observer) are just different frontend views of the SAME data. When the Secretary creates a meeting for BoD, every BoD board member must see it. When minutes are created, they must be linked to their meeting and board. There must be NO orphaned entities. Currently meetings and votes work correctly (linked to board_id 455aebdd), but minutes are created with meetingId: null and boardName: null — making them invisible to everyone except the Secretary.

Fix these in EXACT order. Test each fix with the verification steps provided before moving to the next.

---

## ROUND 1 — CRITICAL DATA LAYER FIXES

### FIX 1: API endpoints must be role-protected (SECURITY)

**Problem:** Board member browser calls /api/pending-actions, /api/people, /api/documents, /api/tasks — all return 200 with full data. Frontend routing protects the UI but API endpoints are wide open.

**Fix:** Add role-based middleware to all endpoints:
- /api/pending-actions/* → admin only
- /api/people → admin sees all; others see only their board's members
- /api/documents → admin sees all; others filtered by board membership
- /api/tasks → admin sees all; management sees only their assigned tasks
- /api/minutes → admin sees all; others filtered by board membership
- /api/meetings → admin sees all; others filtered by board membership
- /api/votes → admin sees all; others filtered by board membership

**Test:** Log in as board member (s.chen@meridian-energy.com / Meridian2024!). Open browser DevTools → Console. Run:
```javascript
fetch('/api/pending-actions?status=pending', { headers: { Authorization: 'Bearer ' + localStorage.getItem('token') } }).then(r => console.log('Status:', r.status))
```
Must return 403, not 200.

### FIX 2: Minutes must be linked to meeting and board on creation

**Problem:** When the AI proposes create_minutes and Secretary approves, the minutes are created with meetingId: null and boardName: null. This makes them invisible in board rooms (board members, management, observers see "No minutes").

**Fix:** In the approval handler for action_type === 'create_minutes':
1. Set minutes.meeting_id from action_data (the AI proposes linking to an existing meeting)
2. If no meeting match, look up the board from the document's AI classification
3. The minutes list queries in board rooms filter by meeting.board_id — so minutes MUST have a meeting_id that links to a meeting with a board_id

Also fix the same issue for create_vote and create_task — ensure board_id is always set.

**Test:** As Secretary, upload a .txt file with board minutes. Approve the "Create Minutes" action. Then run in console:
```javascript
fetch('/api/minutes', { headers: { Authorization: 'Bearer ' + localStorage.getItem('token') } }).then(r => r.json()).then(m => console.log('meetingId:', m[m.length-1].meetingId, 'boardName:', m[m.length-1].boardName))
```
meetingId must NOT be null. Then log in as board member → enter BoD room → minutes tab → the minutes must be visible.

### FIX 3: AI-approved meetings must include agenda items AND attendance

**Problem:** Manual meeting creation correctly inserts agenda items and attendance rows. But AI-approved meetings (via pending_actions) create the meeting with 0 agenda items and 0 attendance. The bug is isolated to the pending action approval handler.

**Fix:** In the approval handler for action_type === 'create_meeting':
1. After inserting the meeting row, iterate over action_data.details.agenda_items and INSERT each into the agenda_items table
2. Query board_memberships for the meeting's board and INSERT attendance rows (status: 'pending') for each member

The manual creation path already does both of these correctly — copy that logic into the approval handler.

**Test:** As Secretary, type in the AI Command bar: "Schedule a BoD meeting for May 15 at 2pm with ESG Review and Budget Approval on the agenda". Approve the pending action. Navigate to /secretary/meetings → click the new meeting. Verify:
- Agenda shows "ESG Review" and "Budget Approval" (not "0 agenda items")
- Attendance shows all 10 BoD members

### FIX 4: Timezone bug on AI-proposed meeting times

**Problem:** AI Command "schedule meeting at 10am" → stored as 4:00 AM. Manual datetime-local input works correctly. Bug is isolated to parsing AI time strings.

**Fix:** When the AI returns a time in action_data (e.g., "10:00 AM" or "2026-04-20T10:00"), treat it as local time. Don't apply UTC conversion. Store directly.

**Test:** As Secretary, use AI Command: "Schedule a BoD meeting for June 1 at 3pm." Approve. Check meetings list — must show 3:00 PM, not a different time.

---

### FIX 5: Duplicate vote returns 500 instead of 409

**Problem:** UNIQUE constraint on vote_records(vote_id, person_id) prevents duplicates (good), but the error isn't caught — returns 500 with HTML error page.

**Fix:** In the vote cast endpoint, catch the unique constraint error and return: `{ status: 409, message: "You have already voted on this resolution" }`

**Test:** Log in as board member. Vote on a resolution. Then run in console:
```javascript
fetch('/api/votes/VOTE_ID/cast', { method: 'POST', headers: { 'Content-Type': 'application/json', Authorization: 'Bearer ' + localStorage.getItem('token') }, body: JSON.stringify({ decision: 'approved' }) }).then(r => console.log(r.status))
```
Must return 409, not 500.

### FIX 6: "Mark Concluded" button is dead

**Problem:** On meeting detail page, the "Mark Concluded" button has no click handler. No API call fires, no console error, no state change.

**Fix:** Wire the button to call PATCH /api/meetings/{id} with { status: 'concluded' }. After success, update the UI to show "Concluded" status and hide the "Mark Concluded" button.

**Test:** As Secretary, go to a meeting detail → click "Mark Concluded." Status must change to "concluded." Meeting list must show "concluded" badge instead of "scheduled."
