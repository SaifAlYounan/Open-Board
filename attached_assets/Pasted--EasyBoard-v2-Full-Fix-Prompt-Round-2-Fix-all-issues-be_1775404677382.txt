## FIX 3 — AI SEARCH PARSING (CRITICAL)

The AI search bar returns "AI response could not be parsed" when board members search. Fix:

**Root cause:** The SEARCH mode prompt tells the AI to return JSON, but the AI is returning natural language text. The frontend then tries to JSON.parse() it and fails.

**Fix option A (recommended):** Change the SEARCH prompt to tell the AI to return plain text, not JSON. Search answers should be natural language with entity references. Don't try to parse as JSON.

```javascript
// In the search endpoint handler:
const response = await callAI('SEARCH', systemPrompt, userContent);
// For SEARCH mode, don't JSON.parse — return the raw text as the answer
if (mode === 'SEARCH') {
  return { success: true, data: { answer: response.content[0].text } };
}
```

**Fix option B:** Update the SEARCH prompt to require JSON output:
```json
{ "answer": "natural language answer", "sources": [{"type": "meeting", "id": "uuid", "title": "..."}] }
```
And update the callAI function to handle SEARCH mode parsing accordingly.

**Frontend:** Display the AI answer as formatted text (support markdown). If sources are returned, render them as clickable links to entity detail pages.

**TEST:** Log in as n.petrov@meridian-energy.com. Type "What meetings are scheduled?" in the AI search bar. Verify a natural language answer appears without errors.

## FIX 4 — TASK ACCESS CONTROL (CRITICAL)

Management users cannot see tasks assigned to them. The /api/tasks endpoint returns an empty array for management users even when tasks exist.

**Fix:** The tasks query must include tasks where:
- `assignee_id = req.user.id` (tasks assigned to this user), OR
- User has access via access_control table, OR
- User role is admin (sees all tasks)

```sql
SELECT t.* FROM tasks t
WHERE t.assignee_id = $1  -- user's person id
   OR EXISTS (SELECT 1 FROM access_control ac WHERE ac.entity_type = 'task' AND ac.entity_id = t.id AND ac.person_id = $1)
   OR $2 = 'admin'  -- admin sees all
ORDER BY t.due_date ASC
```

**Also fix grantDefaultAccess for tasks:** When a task is created (via AI approval or manual), call grantDefaultAccess which should grant access to:
- The assignee
- All admin users
- All observers of the source board

## FIX 5 — PROPER HTTP STATUS CODES (HIGH)

API returns HTTP 200 with error body instead of proper status codes. Fix ALL auth/error responses:

- Invalid/missing JWT token → HTTP 401 `{"error": "Unauthorized"}`
- Valid token but wrong role → HTTP 403 `{"error": "Forbidden"}`
- Resource not found → HTTP 404 `{"error": "Not found"}`
- Duplicate vote → HTTP 409 `{"error": "Already voted"}`
- Validation error → HTTP 400 `{"error": "description"}`
- Server error → HTTP 500 `{"error": "Internal server error"}`

**Fix in auth middleware:**
```javascript
if (!token) return res.status(401).json({ error: 'Unauthorized' });
try {
  const decoded = jwt.verify(token, process.env.JWT_SECRET);
  req.user = decoded;
} catch (err) {
  return res.status(401).json({ error: 'Invalid token' });
}
```

**TEST:** Call any API endpoint with no Authorization header. Verify HTTP 401 (not 200).
