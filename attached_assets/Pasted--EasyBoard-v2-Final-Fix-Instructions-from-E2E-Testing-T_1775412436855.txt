### PRIORITY 6: AI-approved meetings missing agenda items and attendance

**What happens:** When Secretary uses AI Command to schedule a meeting and approves the pending action, the meeting is created with 0 agenda items and 0 attendance rows. Manual meeting creation works correctly — the bug is isolated to the pending action approval handler.

**Fix:** In the approval handler for action_type === 'create_meeting':
1. After inserting the meeting, loop over action_data agenda items and INSERT into agenda_items table
2. Query board_memberships for the meeting's board and INSERT attendance rows with status 'pending'

Copy the logic from the manual meeting creation endpoint — it already does both correctly.

**How to test:** As Secretary, AI Command: "Schedule a BoD meeting for June 1 at 2pm with ESG Review on the agenda" → Approve → open meeting → MUST show agenda item AND 10 BoD members in attendance.

### PRIORITY 7: Duplicate vote returns 500 instead of 409

**What happens:** When a board member tries to vote twice on the same resolution, the server returns HTTP 500 (with an HTML error page) instead of 409 Conflict. The UNIQUE constraint prevents the duplicate, but the error isn't caught.

**Fix:** In the vote cast endpoint, wrap the INSERT in a try/catch. If the error is a unique constraint violation, return `{ status: 409, message: "You have already voted" }`.

**How to test:** Vote on a resolution. Then in browser console:
```javascript
fetch('/api/votes/VOTE_ID/cast', { method: 'POST', headers: { 'Content-Type': 'application/json', Authorization: 'Bearer ' + localStorage.getItem('token') }, body: JSON.stringify({ decision: 'approved' }) }).then(r => console.log(r.status))
```
Must return 409, not 500.

### PRIORITY 8: Timezone on AI-proposed meeting times

**What happens:** AI Command "schedule meeting at 10am" stores the time as 4:00 AM (6 hour offset). Manual datetime-local input works correctly.

**Fix:** When parsing the AI's proposed time string, treat it as local time — don't apply UTC conversion.

### PRIORITY 9: API endpoints not role-protected (SECURITY)

**What happens:** A board member's browser can call /api/pending-actions, /api/people, /api/documents, /api/tasks — all return 200 with full data. The frontend routing protects the UI but the API is wide open.

**Fix:** Add role-checking middleware:
- /api/pending-actions → admin only (403 for others)
- /api/people → admin sees all, others see only their board members
- /api/documents → filtered by board membership
- /api/tasks → admin sees all, management sees only assigned
