## ROUND 1 — CRITICAL BUGS (fix first, everything depends on these)

### BUG 1: API endpoints not role-protected (SECURITY HOLE)
Board member's browser calls /api/pending-actions, /api/people, /api/documents, /api/tasks — all return 200 with full data. Frontend routing protects the UI but API endpoints are wide open.

**Fix:** Add role-based middleware:
- /api/pending-actions/* → admin only
- /api/people → admin only (others see only their board's members)
- /api/documents → filter by access_control
- /api/tasks → admin sees all, management sees assigned only
- /api/minutes → filter by access_control  
- /api/meetings → filter by access_control

### BUG 2: AI-approved meetings missing agenda items AND attendance
Manual meeting creation correctly preserves agenda items and auto-populates attendance from board membership. But AI-approved meetings via pending_actions have 0 agenda items and 0 attendance. The bug is isolated to the pending action approval handler.

**Fix:** In the approval handler for action_type === 'create_meeting':
1. After inserting the meeting row, loop over action_data.details.agenda_items and insert each into agenda_items table
2. Query board_memberships for the meeting's board and insert attendance rows (status: 'pending')
3. Call grantDefaultAccess() (see BUG 3)

### BUG 3: Board members can't see meetings or minutes (but CAN see votes)
Meetings tab in board room: "No meetings for this board." Minutes tab: "No minutes available." Votes work fine. The queries for meetings and minutes are filtering differently than votes.

**Fix:** Either:
- Implement grantDefaultAccess() and create access_control rows when entities are created, then ensure meeting/minutes queries join against access_control
- OR: make meeting/minutes queries filter by board_id + board_membership (same way votes apparently work), without requiring per-entity access_control rows
Both approaches work. Pick one and apply consistently to ALL entity types.

### BUG 5: Tasks may auto-execute without Secretary approval
Tasks TASK-2026-001 and TASK-2026-002 appeared in the Tasks list even though I never explicitly approved the create_task pending actions. Rule #9: "Secretary approval is mandatory — no action executes without approval."

**Fix:** Verify that document upload → AI classification ONLY creates pending_action rows. The execution function should ONLY be called when the Secretary clicks Approve. Check for any code path that calls the execution function directly from the upload/classification handler.

### BUG 6: Timezone bug on AI-proposed meeting times
AI Command specified "10am" but meeting stored as 04:00 AM. Manual datetime-local input works correctly (2:00 PM shows as 2:00 PM). Bug is isolated to parsing AI-proposed time strings.

**Fix:** When the AI returns a time in action_data (e.g., "10:00 AM" or "2026-04-20T10:00:00"), parse it as local time, not UTC. Or store it directly as the AI proposed without timezone conversion.

### BUG 7: Duplicate vote returns 500 instead of 409
The UNIQUE constraint on vote_records(vote_id, person_id) prevents duplicates (good), but the error is not caught — server returns 500 with HTML error page instead of 409 JSON.

**Fix:** In the vote cast endpoint, catch the unique constraint violation error and return { status: 409, message: "You have already voted on this resolution" }.

### MISSING 7: AI Search returns 500

### MISSING 8: AI Insights show stale/contradictory data
Board member dashboard AI Insights said "You have not yet cast your vote on RES-BOD-2026-001" AFTER the vote was already cast. Also said "Minutes Awaiting Your Signature" but Minutes to Sign counter showed 0. The SUGGEST mode is receiving stale data or not checking the user's vote_records/access.

**Fix:** Before calling AI SUGGEST, query vote_records to check if user has already voted, and query access_control/minutes_signatures to check signing status. Pass accurate data to the AI.

### MISSING 9: Pending Actions counter doesn't refresh
After uploading a document that created 4 pending actions, dashboard still showed "Pending AI Actions: 0". 

**Fix:** After document upload + classification, invalidate/refetch the pending actions count query.
